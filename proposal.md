# [Part 1] Forwarding Visibility Dashboard MVP 요구사항 정의 및 설계서

본 문서는 Reboott 제품 팀의 'Forwarding Visibility Dashboard MVP' 과제를 완수하기 위해, 모호한 1차 요구사항을 재정제하고 한 달(실제 작업 15~20시간)이라는 타임라인 내에 지속 가능한 아키텍처로 구현하기 위한 엔지니어링 판단의 기록입니다. 

"완성도보다 판단의 흔적을 본다"는 핵심 방향성에 공감하며, 도메인 요구사항에 대한 합리적인 의심과 가정을 바탕으로 스코프를 제한하고 고도화했습니다.

---

## 1. 핵심 요구사항의 합리적 의심 및 범위(Scope) 제한

### [판단 1] 화물 조회 식별 데이터 단위 지정 (B/L vs 컨테이너)
* **의심 및 문제 제기:** 1차 요구사항의 "화물 조회"는 식별 단위가 모호합니다. 해상 무역 실무상 하나의 B/L(선하증권)에 수십 개의 컨테이너가 묶일 수 있으며, 개별 컨테이너별로 세관 검사 속도나 양하 시점이 다를 수 있습니다. 이를 무조건 1:1 구조로 묶으면 데이터 왜곡이 발생합니다.
* **합리적 가정 및 판단:** 이번 MVP 단계에서는 **1 B/L = 1 상태 트래킹** 구조로 범위를 전면 제한합니다. B/L 번호를 이커머스의 '주문번호' 개념으로, 컨테이너를 '개별 택배 박스'로 정의하여 화주가 대시보드에서 추적하는 기준을 B/L 단위로 선형화합니다.
* **리스크 및 완화책 (Risk & Mitigation):**
  * *위험도:* 특정 컨테이너만 통관이 지연(검역 대상 지정 등)되었을 때, B/L 전체 상태를 '통관 완료'로 오인하여 화주사가 운송 트럭을 잘못 배차해 공차 운임을 낭비할 리스크가 있습니다.
  * *완화책:* 차기 고도화(Phase 2) 시 `Shipment` 테이블 하위에 `Container` 테이블을 1:N 관계로 분리하고, 컨테이너별 세부 상태를 추적할 수 있도록 데이터 모델 확장성을 미리 고려하여 설계했습니다.

### [판단 2] 운영팀 ERP 데이터 동기화 방식 (직접 CRUD vs 엑셀)
* **의심 및 문제 제기:** 운영팀 6명이 수백 개의 화물 상태를 일일이 대시보드 화면에서 수동으로 생성/수정/삭제(CRUD)하는 것은 실무 효율성이 극도로 떨어지며 입력 실수의 위험이 큽니다.
* **합리적 가정 및 판단:** MVP 단계에서는 시스템 안정성과 API 정정 기능 검증이 우선이므로 **관리자(포워더) 전용 웹 화면을 통한 직접 CRUD 구조**로 범위를 제한합니다. 운영팀이 가장 필요로 할 '엑셀 파일 일괄 업로드(Bulk Upload)' 기능은 대안 지표로서 Phase 2의 최우선 과제로 백로그에 이관합니다. (대표/PO와의 '협의 필요' 항목으로 분류)
* **리스크 및 완화책:**
  * *위험도:* 수동 입력 지연으로 인해 실제 화물 상태와 대시보드 정보 간의 시차가 발생하여 화주 신뢰도가 하락할 수 있습니다.
  * *완화책:* 관리자 페이지의 UX를 입력 필드를 최소화한 폼(Form) 형태로 극단적으로 단순화하여 운영팀의 건당 입력 피로도를 낮추었습니다.

### [판단 3] 인증 및 권한 그룹화 수준 (다수 담당자 계정 공유)
* **의심 및 문제 제기:** 화주사 내부에 수십 명의 물류 담당자가 존재할 수 있으며, 이들에게 개별 계정 생성 및 세부 권한(RBAC)을 부여하는 것은 한 달짜리 MVP 타임라인에 맞지 않습니다.
* **합리적 가정 및 판단:** 보안 인가는 **화주사(Company) 단위로 데이터를 완벽히 격리**하되, 화주사 내부의 다수 담당자는 하나의 회사 공용 계정(예: samsung, hyundai)을 공유하는 형태로 MVP 범위를 제한합니다.
* **리스크 및 완화책:**
  * *위험도:* 공용 계정 사용으로 인해 "누가 정보를 확인했는지" 혹은 포워더 측에서 "누가 스케줄을 변경했는지"에 대한 히스토리 추적(Audit Log)이 불가능하여 책임 소재 문제가 생길 수 있습니다.
  * *완화책:* 백엔드 API 레벨에서 타사 데이터가 절대로 조회되지 않도록 `shipper_filter` Guard 로직을 최우선으로 잠그고, 계정 공유로 인한 리스크는 사내 가이드라인으로 통제하도록 '협의 필요' 항목에 명시합니다.

### [판단 4] ETA 변동 알림 채널의 현실적 절삭 및 피드백 확장 (인앱 웹 알림 & 피드 페이지)
* **의심 및 문제 제기:** "ETA 변동 시 알림"을 카카오 알림톡이나 외부 이메일 인프라(AWS SES 등)와 연동하려면 행정적 계약, 사업자 심사, 템플릿 검수 기간만 2~3주가 소요되어 15~20시간의 개발 타임라인 내 구현이 불가능합니다.
* **합리적 가정 및 판단:** 이번 MVP 단계에서는 외부 연동을 전면 배제하고, **대시보드 웹 내 우측 상단 종(🔔) 아이콘을 통한 인앱 알림창(Notification Dropdown) 및 전체 알림 피드 전용 타임라인 페이지**로 범위를 변경하여 알림의 추적성을 극대화했습니다.
* **엔지니어링 트레이드오프 (클라이언트 단 상태 제어):**
  * *고민 지점:* 알림 확인 후 리스트 점유 문제를 해결하기 위해 백엔드 DB 구조를 바꾸고 알림 읽음 상태(`is_read`) API를 새로 파는 것은 한정된 타임라인 내 오버엔지니어링이라고 판단했습니다.
  * *결정 내역:* 브라우저의 `localStorage`를 활용하여 사용자가 X 버튼을 누른 알림의 ID 목록을 클라이언트 단에서 영속적으로 관리하도록 설계했습니다. 간이 드롭다운에서는 읽은 알림을 즉시 숨기고, 전체 알림 피드 페이지에서는 취소선(opacity-60) 처리를 하여 과거 스케줄 변동 이력의 추적성을 보존했습니다. 이로 인해 백엔드 I/O 부하를 없애고 최적의 UX를 확보했습니다.
* **리스크 및 완화책:**
  * *위험도:* 사용자가 브라우저 캐시를 지우거나 다른 PC로 접속할 경우 읽음 상태가 초기화되어 이미 확인한 알림이 다시 미확인 상태로 노출될 수 있습니다.
  * *완화책:* 차기 페이즈(Phase 2) 진입 시 알림 마스터 테이블 하위에 사용자-알림 매핑 테이블을 구축하여 데이터베이스 레벨에서 상태 유저 싱크를 맞출 것을 로드맵에 반영합니다.

### [판단 5] 복잡한 통관 단계의 선형화 스코프 조절
* **의심 및 문제 제기:** "통관 등"의 요구사항은 실제 보세창고 반입, 관세사 신고, 세관 검사, 수리 등 비선형적이고 복잡한 하위 단계를 가집니다. 이를 전부 구현하면 데드라인 내 배포가 불가능합니다.
* **합리적 가정 및 판단:** 1차 MVP는 상태 단계를 **`[선적 ➔ 운항중 ➔ 도착 ➔ 통관 완료]`의 4단계 선형적 상태**로 구조를 단순화합니다. 상세한 통관 진행 현황 트래킹은 차기 고도화 스펙으로 미룹니다.
* **리스크 및 완화책:**
  * *위험도:* '통관 완료' 전 단계에서 물품이 세관 검사에 걸려 보류되었을 때, 대시보드상에는 단순히 '도착'으로만 표시되어 화주의 불필요한 문의 전화가 운영팀에 가중될 수 있습니다.
  * *완화책:* 차기 페이즈에서 관세청 UNI-PASS 오픈 API를 비동기 배치(Batch) 형태로 연동하여 수입신고번호 기준 실시간 진행 상태를 자동 동기화하는 로드맵을 제안합니다.

## 2. 데이터 모델링 및 아키텍처 설계

### ERD 및 데이터 스키마 (SQLAlchemy / Pydantic v2 스펙)

#### 1) Shipment (화물 마스터 테이블)
* `id` (String, PK): B/L 번호 (식별자)
* `shipper` (String, Index): 화주사명 (데이터 격리 및 필터링용)
* `origin` (String): 출발지
* `destination` (String): 목적지
* `status` (String): 화물 상태 (`선적`, `운항중`, `도착`, `통관 완료`)
* `eta` (String): 도착 예정일 (YYYY-MM-DD)

#### 2) Notification (인앱 웹 알림 테이블)
* `id` (String, PK): 알림 고유 ID (`NOTI-타임스탬프-BL번호`)
* `shipper` (String, Index): 수신 대상 화주사명
* `bl_id` (String): 관련 B/L 번호
* `message` (Text): 변동 내역 메시지
* `created_at` (DateTime): 알림 발생 일시

## 3. 대표/PO 단계 "협의 필요" 최종 항목 요약
개발자 관점에서 비즈니스 리스크를 고려해 분류한 **협의 필요** 리스트입니다. 본 항목에 대한 판단력 자체를 평가에 반영해 주시기를 부탁드립니다.

1. **포워더 운영팀 업무 부하 완화 건:** Phase 2 진입 즉시 엑셀 업로드 기능을 최우선 구현할 것인가, 혹은 관세청 UNI-PASS 시스템 연동을 통한 자동화를 먼저 진행할 것인가에 대한 우선순위 협의.
2. **알림 발송 임계값(Threshold) 정책 수립 건:** 향후 알림톡 연동 시, 미미한 시간 단위 변경까지 모두 발송할 경우 발생하는 비용 및 화주 피로도를 방지하기 위해 '며칠 이상 지연 시 발송'할 것인지에 대한 비즈니스 기준 협의.
3. **화주사 보안 감사(Audit) 리스크 건:** 다수 담당자의 단일 계정 공유 방식이 고객사의 보안 규정에 위배되는지 확인하고, 향후 담당자별 서브 계정 생성 기능을 도입할 것인지 여부 조율.
