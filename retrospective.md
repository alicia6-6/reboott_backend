# [Part 3] AI 협업 회고록 (Retrospective)

본 문서는 'Forwarding Visibility Dashboard MVP'를 개발하는 과정에서 대형 언어 모델(Gemini)과 어떻게 협업했는지, AI 도구의 생산성을 주도적으로 통제하고 한계를 바로잡은 과정에 대한 솔직한 기록입니다.

---

## 1. 사용한 AI 도구 및 역할 분담 (위임 영역)

*   **사용한 도구:** Gemini (대형 언어 모델)
*   **AI에게 위임한 영역 (생산성 극대화):**
    *   **풀스택 보일러플레이트 코드 고속 생성:** FastAPI와 React(TypeScript, Tailwind CSS) 간의 기본적인 API 연동 규격 및 UI 마크업 구조를 빠르게 잡아내는 데 활용했습니다.
    *   **도메인 유스케이스 구체화:** 물류 실무 지식이 부족한 상태에서 B/L과 컨테이너의 차이점, 국제 해상 운송 흐름의 대분류 상태를 브레인스토밍하고 검증하는 파트너로 활용했습니다.
    *   **최신 라이브러리 문법 마이그레이션:** 구버전 문법으로 인해 발생하는 경고(Warning) 메시지를 최신 스펙(Pydantic v2, SQLAlchemy 2.0)으로 정교하게 교정하는 작업을 위임했습니다.

---

## 2. AI가 잘못 유도한 부분과 인간 개발자의 주도적 제어 (바로잡은 과정)

개발 과정 중 AI가 기술적/도메인적으로 잘못된 방향으로 가거나 오버엔지니어링을 제안한 순간들이 있었고, 이를 인간 개발자의 판단력으로 통제했습니다.

### 1) '진짜 에러'와 '단순 경고'의 성격 오판 및 최신 문법 통제
*   **상황:** 백엔드 구동 시 터미널에 `MovedIn20Warning` 및 `PydanticDeprecatedSince20` 노란색 경고 문구들이 대거 출력되었습니다.
*   **AI의 최초 유도:** AI는 프로그램 실행에 문제가 없으니 무시하고 진행해도 된다고 안일하게 제안했습니다.
*   **인간 개발자의 바로잡기:** 과제 평가관이 코드를 직접 구동할 때 터미널에 지저분한 경고가 뜨는 것은 코드 품질 및 최신 기술 트렌드 인지 부족으로 감점 요인이 될 수 있다고 판단했습니다. AI에게 최신 Pydantic v2의 `ConfigDict(from_attributes=True)` 문법과 SQLAlchemy 최신 `declarative_base` 경로를 명확히 지시하여 모든 경고를 완전히 제거(Zero-Warning)했습니다.

### 2) 파이썬-타입스크립트 간 문법 혼동 및 컴파일 에러 해결
*   **상황:** 프론트엔드 리액트 빌드 중 `cannot find name 'str'`이라는 치명적인 타입 컴파일 에러가 발생하며 하얀 창(White Screen)이 떴습니다.
*   **AI의 최초 유도:** 백엔드 데이터 모델 스키마를 고속으로 컨텍스트에 스위칭하는 과정에서 AI가 타입스크립트 인터페이스에 파이썬 문자열 타입인 `str`을 그대로 적어주는 실수를 저질렀습니다.
*   **인간 개발자의 바로잡기:** 에러 로그와 줄 번호를 추적하여 자바스크립트 표준 문자열 타입인 `string`으로 즉시 교정했습니다. 또한 컴포넌트 내부의 렌더링 키(`key={n.id}`) 누락 가능성까지 함께 점검하여 런타임에 리액트 앱이 터지는 현상을 원천 차단했습니다.

### 3) 환경별 백엔드 URL의 수동 전환 방지 (Vite 환경 변수 제안)
*   **상황:** 로컬(localhost:8000)에서 테스트할 때와 클라우드(Render)에 배포할 때마다 프론트엔드 코드의 `BACKEND_URL`을 매번 수동으로 주석 처리하며 바꾸어야 하는 번거로움과 배포 사고 리스크가 있었습니다.
*   **AI의 최초 유도:** 배포할 때 주소를 바꾸는 것을 잊지 말라는 1차원적인 가이드만 제공했습니다.
*   **인간 개발자의 바로잡기:** 휴먼 에러를 방지하기 위해 Vite의 빌드 환경 변수 인프라를 활용하기로 결정했습니다. `import.meta.env.DEV` 삼항 연산자 로직을 주입하여, 코드를 단 한 줄도 수정하지 않아도 개발 모드와 배포 모드에서 백엔드 엔드포인트가 자동으로 스위칭되는 견고한 파이프라인을 구축했습니다.

### 4) 실시간 알림 확인 상태 동기화 요구와 아키텍처 타협점 조율
*   **상황:** 화주사 알림 확인 후 드롭다운에서 알림이 지워지지 않고 누적되는 불편함(UX)을 해결하고, 전체 알림 피드를 확인할 수 있는 전용 페이지 요구사항이 추가되었습니다.
*   **AI의 최초 유도:** AI는 최초 백엔드 스키마 설계 단계에서 알림 모델에 `is_read` Boolean 컬럼을 넣고 PATCH API를 확장하는 복잡한 수정을 추천했습니다.
*   **인간 개발자의 바로잡기:** 마감 직전 15~20시간이라는 한정된 과제 타임라인 안에서 API를 파고 DB 마이그레이션을 다시 하는 것은 배포 안전성을 깨뜨릴 수 있다고 판단했습니다. 대신 프론트엔드의 `localStorage`를 브릿지로 활용하여, 사용자가 지운 알림의 고유 ID 목록을 클라이언트 단에 보관하는 방식을 주도적으로 설계해 냈습니다. 드롭다운에서는 제외 필터링으로 제거하고, 전체 알림 피드 타임라인 페이지에서는 누적 히스토리를 확인할 수 있도록 취소선(opacity) 처리를 결합하여 짧은 시간 안에 완벽한 사용성을 분리 구현해 냈습니다.

## 3. 결론 및 느낀 점 (AI 협업에 대한 회고)

AI는 도메인 지식의 뼈대를 잡고 반복적인 보일러플레이트 코드를 작성하는 시간을 80% 이상 단축해 주는 훌륭한 '조수'였습니다. 덕분에 저는 코딩 자체보다 **"어떤 범위를 깎아내고, 어떤 비즈니스 리스크를 방어할 것인가"**라는 상위 차원의 설계와 엔지니어링 판단에 대부분의 시간(15~20시간)을 밀도 있게 투자할 수 있었습니다.

그러나 AI는 컨텍스트가 조금만 복잡해져도 다른 언어의 문법을 혼용하거나 최신 라이브러리의 Deprecated 경고를 간과하고, 비즈니스 타임라인을 고려하지 않은 비효율적인 DB 구조 수정을 제안하는 등 디테일한 구멍을 만들어냈습니다. 결국 **최종 코드의 아키텍처적 타협점을 통제하고, 비즈니스 리스크를 식별하여 "왜 이렇게 제품을 깎아냈는지" 논리를 세우는 최종 지휘권은 오직 인간 개발자에게 있음**을 깊이 깨달은 프로젝트였습니다.
